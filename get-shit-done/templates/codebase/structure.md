<good_examples>
```markdown
# Codebase Structure

**วันที่วิเคราะห์:** 2025-01-20

## Directory Layout

```
get-shit-done/
├── bin/                # Executable entry points
├── commands/           # Slash command definitions
│   └── gsd/           # GSD-specific commands
├── get-shit-done/     # Skill resources
│   ├── references/    # Principle documents
│   ├── templates/     # File templates
│   └── workflows/     # Multi-step procedures
├── src/               # Source code (ถ้ามี)
├── tests/             # Test files
├── package.json       # Project manifest
└── README.md          # User documentation
```

## Directory Purposes

**bin/**
- วัตถุประสงค์: CLI entry points
- ประกอบด้วย: install.js (installer script)
- ไฟล์สำคัญ: install.js - handles npx installation
- Subdirectories: ไม่มี

**commands/gsd/**
- วัตถุประสงค์: Slash command definitions สำหรับ Claude Code
- ประกอบด้วย: ไฟล์ *.md (หนึ่งต่อ command)
- ไฟล์สำคัญ: new-project.md, plan-phase.md, execute-plan.md
- Subdirectories: ไม่มี (flat structure)

**get-shit-done/references/**
- วัตถุประสงค์: Core philosophy และ guidance documents
- ประกอบด้วย: principles.md, questioning.md, plan-format.md
- ไฟล์สำคัญ: principles.md - system philosophy
- Subdirectories: ไม่มี

**get-shit-done/templates/**
- วัตถุประสงค์: Document templates สำหรับ .planning/ files
- ประกอบด้วย: Template definitions พร้อม frontmatter
- ไฟล์สำคัญ: project.md, roadmap.md, plan.md, summary.md
- Subdirectories: codebase/ (ใหม่ - สำหรับ stack/architecture/structure templates)

**get-shit-done/workflows/**
- วัตถุประสงค์: Reusable multi-step procedures
- ประกอบด้วย: Workflow definitions ที่เรียกโดย commands
- ไฟล์สำคัญ: execute-phase.md, research-phase.md
- Subdirectories: ไม่มี

## Key File Locations

**Entry Points:**
- `bin/install.js` - Installation script (npx entry)

**Configuration:**
- `package.json` - Project metadata, dependencies, bin entry
- `.gitignore` - Excluded files

**Core Logic:**
- `bin/install.js` - ทุก installation logic (file copying, path replacement)

**Testing:**
- `tests/` - Test files (ถ้ามี)

**Documentation:**
- `README.md` - User-facing installation และ usage guide
- `CLAUDE.md` - Instructions สำหรับ Claude Code เมื่อทำงานใน repo นี้

## Naming Conventions

**Files:**
- kebab-case.md: Markdown documents
- kebab-case.js: JavaScript source files
- UPPERCASE.md: Important project files (README, CLAUDE, CHANGELOG)

**Directories:**
- kebab-case: ทุก directories
- Plural สำหรับ collections: templates/, commands/, workflows/

**Special Patterns:**
- {command-name}.md: Slash command definition
- *-template.md: ใช้ได้แต่ templates/ directory ถูกต้องกว่า

## Where to Add New Code

**Slash Command ใหม่:**
- Primary code: `commands/gsd/{command-name}.md`
- Tests: `tests/commands/{command-name}.test.js` (ถ้า testing implemented)
- Documentation: อัปเดต `README.md` พร้อม command ใหม่

**Template ใหม่:**
- Implementation: `get-shit-done/templates/{name}.md`
- Documentation: Template เป็น self-documenting (รวม guidelines)

**Workflow ใหม่:**
- Implementation: `get-shit-done/workflows/{name}.md`
- Usage: Reference จาก command พร้อม `@~/.claude/get-shit-done/workflows/{name}.md`

**Reference Document ใหม่:**
- Implementation: `get-shit-done/references/{name}.md`
- Usage: Reference จาก commands/workflows ตามต้องการ

**Utilities:**
- ยังไม่มี utilities (`install.js` เป็น monolithic)
- ถ้า extracted: `src/utils/`

## Special Directories

**get-shit-done/**
- วัตถุประสงค์: Resources ที่ installed ไปยัง ~/.claude/
- Source: Copied โดย bin/install.js ระหว่าง installation
- Committed: Yes (source of truth)

**commands/**
- วัตถุประสงค์: Slash commands ที่ installed ไปยัง ~/.claude/commands/
- Source: Copied โดย bin/install.js ระหว่าง installation
- Committed: Yes (source of truth)
```

<guidelines>
**อะไรควรอยู่ใน STRUCTURE.md:**
- Directory layout (ASCII tree)
- วัตถุประสงค์ของแต่ละ directory
- Key file locations (entry points, configs, core logic)
- Naming conventions
- ที่จะเพิ่ม code ใหม่ (แยกตามประเภท)
- Special/generated directories

**อะไรไม่ควรอยู่ที่นี่:**
- Conceptual architecture (นั่นคือ ARCHITECTURE.md)
- Technology stack (นั่นคือ STACK.md)
- Code implementation details (defer ไปอ่านโค้ด)
- ทุกไฟล์เดี่ยวๆ (เน้นที่ directories และ key files)

**เมื่อกรอกเทมเพลตนี้:**
- ใช้ `tree -L 2` หรือคล้ายกันเพื่อ visualize structure
- ระบุ top-level directories และวัตถุประสงค์
- Note naming patterns โดยสังเกต files ที่มีอยู่
- ระบุ entry points, configs และ main logic areas
- รักษา directory tree ให้กระชับ (max 2-3 levels)

**รูปแบบ ASCII tree:**
```
root/
├── dir1/           # วัตถุประสงค์
│   ├── subdir/    # วัตถุประสงค์
│   └── file.ts    # วัตถุประสงค์
├── dir2/          # วัตถุประสงค์
└── file.ts        # วัตถุประสงค์
```

**มีประโยชน์สำหรับ phase planning เมื่อ:**
- เพิ่ม features ใหม่ (ไฟล์ควรไปที่ไหน?)
- เข้าใจ project organization
- หาว่า specific logic อยู่ที่ไหน
- ทำตาม existing conventions
</guidelines>
